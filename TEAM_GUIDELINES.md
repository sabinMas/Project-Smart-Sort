# Team Guidelines - Box Smart Inbox

Central rules for all team members to ensure parallel work without conflicts.

## Architecture Overview

Three independent domains that communicate through **locked type contracts**:

```
Email → [Domain 1: Ingestion] → [Domain 2: Classification] → [Domain 3: Box] → Notifications
```

**Key Principle:** Each domain is completely independent. Domains only talk via shared types.

---

## Type Contracts (LOCKED - DO NOT MODIFY)

Located in `backend/shared/types.py`. These are the boundaries between domains.

### IngestedDocument
**Output of Domain 1, Input to Domain 2**
- Generated by: Email ingestion service
- Consumed by: Classification service
- Fields: id, filename, content, content_type, uploaded_at, source, email_from, file_size_bytes

### ClassificationResult  
**Output of Domain 2, Input to Domain 3**
- Generated by: Classification service
- Consumed by: Box integration service
- Fields: document_id, doc_type, confidence, reasoning, extracted_fields, required_reviewer, metadata_tags, classified_at

### ProcessingResult
**Output of Domain 3, sent to notifications/API**
- Generated by: Box integration service
- Consumed by: main.py, API clients
- Fields: document_id, box_file_id, destination_folder, status, task_id, assigned_to, metadata_applied, notification_sent_to, error_message, completed_at

**Rule:** These types CANNOT be changed after T+2h without ALL team members' agreement.

## Domain Ownership

### Domain 1: Email Ingestion (Person A)
- **Files you own:** `backend/domain_1_email/` (all files in this directory)
- **Input:** Email from SendGrid webhook
- **Output:** `IngestedDocument` object
- **Responsibility:** Extract email content, attachments, validate, return structured document
- **Estimated time:** 3-4 hours
- **Critical success:** Return IngestedDocument with `content` field fully populated

### Domain 2: AI Classification (Person B)
- **Files you own:** `backend/domain_2_classifier/` (all files in this directory)
- **Input:** `IngestedDocument` object
- **Output:** `ClassificationResult` object
- **Responsibility:** Call LLM, parse response, validate, return classification
- **Estimated time:** 3-4 hours
- **Critical success:** Confidence score must be float 0.0-1.0, doc_type must be valid

### Domain 3: Box Integration (Person C)
- **Files you own:** `backend/domain_3_box_integration/` (all files in this directory)
- **Input:** `ClassificationResult` object
- **Output:** `ProcessingResult` object
- **Responsibility:** Route to Box folder, apply metadata, create task, send notifications
- **Estimated time:** 4-5 hours
- **Critical success:** File correctly placed in Box, task assigned to right reviewer, notifications sent

### Product Owner: You
- **Files you own:** `backend/main.py`, `backend/shared/`, integration tests
- **Responsibility:** Orchestrate domains, manage merges, make decisions
- **Role:** Unblock team, verify integration, demo preparation

## Strict Rules for Code Quality

### Rule 1: No Cross-Domain Imports

```python
# ✅ ALLOWED
from backend.shared.types import IngestedDocument
from backend.domain_1_email.service import EmailIngestionService
from backend.shared.config import Config

# ❌ FORBIDDEN (Domain boundaries violated)
# Domain 1 importing from Domain 2
from backend.domain_2_classifier import ClassificationService

# Domain 2 importing from Domain 3  
from backend.domain_3_box_integration import BoxIntegrationService

# Domain 3 importing from Domain 1
from backend.domain_1_email import EmailIngestionService
```

**Why?** Cross-domain imports cause:
- Hidden dependencies that lead to conflicts
- Circular import errors
- Merge conflicts when domains change
- Harder to test domains independently

**How to stay compliant:**
- Only import from your domain and `shared/`
- Use type contracts (`IngestedDocument`, etc.) as your interface
- Let main.py orchestrate the domains

### Rule 2: Only Edit Your Files

**Domain 1 (Person A):**
```
✅ backend/domain_1_email/**
❌ backend/domain_2_classifier/**
❌ backend/domain_3_box_integration/**
❌ backend/main.py (unless you're the PO)
```

**Domain 2 (Person B):**
```
✅ backend/domain_2_classifier/**
❌ backend/domain_1_email/**
❌ backend/domain_3_box_integration/**
❌ backend/main.py (unless you're the PO)
```

**Domain 3 (Person C):**
```
✅ backend/domain_3_box_integration/**
❌ backend/domain_1_email/**
❌ backend/domain_2_classifier/**
❌ backend/main.py (unless you're the PO)
```

**Exception:** All can modify `backend/shared/` but only for:
- Adding tests
- Adding documentation
- Adding custom exceptions (in errors.py)
- NOT modifying types.py after T+2h

### Rule 3: Test Your Domain in Isolation

```bash
# Domain 1: Run only domain 1 tests
pytest backend/domain_1_email/tests/ -v

# Domain 2: Run only domain 2 tests
pytest backend/domain_2_classifier/tests/ -v

# Domain 3: Run only domain 3 tests
pytest backend/domain_3_box_integration/tests/ -v

# PO: Run integration tests
pytest backend/tests/test_integration.py -v
```

**Each domain should:**
- Have unit tests in `domain_X/tests/`
- Use mocks for other domains
- Use fixtures from `shared/fixtures.py`
- Test independently (don't call other domains' code)

### Rule 4: Commit Messages

Format: `[domain-X] Description of change`

```bash
git commit -m "[domain-1] Add SendGrid webhook signature validation"
git commit -m "[domain-2] Implement Groq LLM fallback"
git commit -m "[domain-3] Create Box task assignment logic"
git commit -m "[shared] Add logging configuration"
```

**Why?** Makes it clear which domain changed and easy to track progress.

### Rule 5: Type Contracts Are Sacred

After T+2h, `backend/shared/types.py` is **LOCKED**.

**If you need more fields:**
```python
# ❌ WRONG: Modify shared type
class IngestedDocument:
    id: str
    new_field_for_domain_1: str  # NO!

# ✅ CORRECT: Create domain-specific model
# In domain_1_email/models.py
class EmailMetadata:
    headers: dict
    from_domain: str  # Domain-specific
    custom_field: str
```

Then pass both the type contract AND your custom data:
```python
document = IngestedDocument(...)  # Required type
metadata = EmailMetadata(...)      # Your extras
return document, metadata
```

## Testing Strategy

### Unit Tests (Per Domain)

**Domain 1: Email Ingestion Tests**
```python
# domain_1_email/tests/test_service.py
def test_ingest_email_with_pdf():
    service = EmailIngestionService()
    result = service.ingest_email(...)
    
    assert isinstance(result, IngestedDocument)
    assert result.content_type == "application/pdf"
    # DON'T test classification or Box operations
```

**Domain 2: Classification Tests**
```python
# domain_2_classifier/tests/test_service.py
async def test_classify_invoice():
    service = ClassificationService()
    result = await service.classify(MOCK_INGESTED_DOCUMENT_INVOICE)
    
    assert isinstance(result, ClassificationResult)
    assert result.doc_type == "invoice"
    # DON'T test email or Box operations
```

**Domain 3: Box Integration Tests**
```python
# domain_3_box_integration/tests/test_service.py
async def test_process_classification():
    service = BoxIntegrationService()
    result = await service.process(MOCK_CLASSIFICATION_INVOICE)
    
    assert isinstance(result, ProcessingResult)
    assert result.status == "success"
    # DON'T test email or classification
```

### Integration Tests (PO)

```python
# backend/tests/test_integration.py
async def test_end_to_end_flow():
    # Domain 1 → Domain 2 → Domain 3
    email = MOCK_INGESTED_DOCUMENT_INVOICE
    classification = await classification_service.classify(email)
    result = await box_service.process(classification)
    
    assert result.status == "success"
    # Test the complete pipeline
```

## Communication & Blocking

### When You're Blocked

**You don't have X, need it from Domain Y:**

❌ **DON'T:** Start coding around the blocker
✅ **DO:** Tell the team immediately

```
@team-pocs: Domain 2 needs Domain 1 to populate 
IngestedDocument.content field completely. 
Currently getting empty strings.
Can you check email text extraction?
```

### When Someone Blocks You

**Domain 1 is waiting on Domain 2 to finish:**

❌ **DON'T:** Ignore the request
✅ **DO:** Prioritize unblocking them

Example response:
```
Thanks for the heads up! I'm on it.
Will have ClassificationResult ready by 2pm.
In the meantime, I'll test with mock data.
```

## Git Workflow

### Before Starting

1. Create your feature branch:
   ```bash
   git checkout -b feature/domain-1-email-ingestion
   git push -u origin feature/domain-1-email-ingestion
   ```

2. Track progress on your branch:
   ```bash
   git add domain_1_email/
   git commit -m "[domain-1] Initial webhook skeleton"
   git push origin feature/domain-1-email-ingestion
   ```

### Before Submitting PR

1. Run your domain tests:
   ```bash
   pytest domain_1_email/ -v
   ```

2. Run ALL tests to ensure no breakage:
   ```bash
   pytest -v
   ```

3. Verify no cross-domain imports:
   ```bash
   grep -r "from backend.domain_" domain_1_email/
   # Should return: 0 matches
   ```

### PR Checklist

- [ ] Only files from your domain modified
- [ ] All tests passing
- [ ] Commit messages follow convention `[domain-X]`
- [ ] No changes to `types.py` (after T+2h)
- [ ] No cross-domain imports
- [ ] Updated domain-specific tests

## Common Mistakes & How to Avoid Them

### Mistake 1: Modifying Another Domain's Files

**How to detect:** 
```bash
git diff main..your-branch | grep "domain_2_classifier"
# If you're not Person B, this should be empty
```

**How to fix:**
```bash
git reset HEAD <wrong-file>
git checkout -- <wrong-file>
```

### Mistake 2: Importing From Another Domain

**How to detect:**
```bash
grep "from backend.domain_" domain_1_email/
# Should be 0 results
```

**How to fix:**
- Remove the import
- Use `shared/types.py` contracts instead
- Pass data through main.py

### Mistake 3: Modifying types.py

**How to detect:**
```bash
git diff main..your-branch -- backend/shared/types.py
# Should be empty
```

**How to fix:**
- Revert: `git checkout main -- backend/shared/types.py`
- Create domain-specific models instead
